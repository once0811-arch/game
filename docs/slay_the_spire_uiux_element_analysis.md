## 아트디렉터 관점 재검토

이 문서는 개발자가 UI를 고치는 기준으로는 유효하지만, 아트디렉터가 바로 제작 판단을 내리기에는 그대로 충분하지 않다. Slay the Spire를 참고한다는 말은 카드 모양이나 양피지 색을 베낀다는 뜻이 아니라, **화면 전체가 플레이 결정을 돕도록 아트 우선순위를 짜는 방식**을 배운다는 뜻이어야 한다.

### 현재 문서가 충분히 말한 것

- 어떤 정보가 어느 화면 위치에 있어야 하는지.
- 지도, 전투, 보상, 상점, 여관, 이벤트가 각각 어떤 결정을 요구하는지.
- 텍스트 버튼, 개발용 패널, 긴 로그, 깨진 카드 배치가 왜 최종 UI로 부적합한지.
- 우리 게임에 없는 Slay 요소를 동료, 서약 전술, 유대, 장비, 증표로 대체해야 한다는 점.

### 현재 문서가 아트 제작 지시로는 부족한 것

- 각 화면에서 실제로 그려야 할 에셋 목록이 바로 나오지 않는다.
- 배경, UI 프레임, 캐릭터, 적, 카드, 아이콘, VFX의 우선순위가 분리되어 있지 않다.
- “예쁘다”의 기준이 화면 합성 기준이 아니라 요소별 설명에 머문다.
- Slay의 화면 문법을 우리 세계관 물성으로 어떻게 번역할지 충분히 못 박지 않았다.
- 작은 화면에서 읽히는 실루엣, 대비, 여백, 텍스트 안전 영역 기준이 더 필요하다.
- 정지 이미지뿐 아니라 hover, drag, target, hit, reward 같은 상태 변화 시안이 필요하다는 말이 약하다.

### 아트디렉터가 이 문서에서 뽑아야 하는 핵심 질문

아트디렉터는 이 분석을 읽고 아래 질문에 답할 수 있어야 한다.

```txt
전투 화면의 첫 시선은 어디에 꽂히는가?
손패가 화면의 주인공으로 보이는가?
적 intent가 일러스트보다 먼저 읽히는가?
동료가 장식 초상이 아니라 이번 런의 전술 자산으로 보이는가?
지도 노드는 텍스트 없이도 의미가 보이는가?
보상 화면은 전리품을 고르는 감각이 있는가?
상점/여관/이벤트는 단순 메뉴가 아니라 장소처럼 보이는가?
증표, 피 지문, 금속패 물성이 UI 전체를 묶고 있는가?
```

### 우리 게임식 시각 번역 원칙

| Slay에서 배울 점 | 우리 게임식 번역 | 아트 실패 기준 |
|---|---|---|
| 전투는 손패와 적 의도를 비교하는 화면 | 길 위 전투 무대, 하단 손패, 적 머리 위 의도, 동료 전술 패널 | 캐릭터 일러스트는 큰데 카드/의도가 안 읽힘 |
| 지도는 미래 선택을 계획하는 화면 | 죽은 지휘관의 작전서, 보급로, 증표 계약 경로 | 노드가 텍스트 버튼처럼 보임 |
| 보상은 덱을 바꾸는 의식 | 전투 후 전리품, 카드 3장, 장비/골드/유대 변화 | 보상이 목록 UI처럼 보임 |
| 카드 프레임은 정보 기계 | 비용, 소유자, 타입, 강화, 희귀도가 작은 크기에서 판독 | 카드 일러스트가 텍스트와 비용을 잡아먹음 |
| UI 물성은 반복 경험을 묶는다 | 금속패, 피 지문, 작전서 종이, 녹슨 장비, 모닥불빛 | 화면마다 서로 다른 게임처럼 보임 |
| 애니메이션은 입력 확인이다 | 카드 hover, 드래그 선, 표식 고정, 금속 반응, 짧은 피해 숫자 | 화려하지만 결과 위치를 모름 |

### 아트 제작 순서에 대한 결론

아트는 개별 PNG를 먼저 예쁘게 만드는 순서로 가면 안 된다. 먼저 필요한 것은 **화면 합성 기준 시안**이다.

1. 전투 화면 컬러 러프: 배경, 플레이어, 적, 손패, 에너지, End Turn, 동료 패널, intent까지 한 장에 배치.
2. 지도 화면 컬러 러프: 작전서 배경, 점선 경로, 노드 아이콘, 선택 가능/불가 상태, 범례, hover 설명.
3. 카드 프레임 시안: 주인공/동료, 공격/스킬/파워, 비용, 강화, 희귀도가 들어간 실제 텍스트 카드.
4. 보상 화면 시안: 카드 3장, 골드, 장비, 스킵, 획득 피드백.
5. 상점/여관/이벤트 시안: 장소감은 있지만 클릭할 선택지가 먼저 읽히는 구조.

즉, 아트디렉터에게 필요한 기준은 “멋진 캐릭터 한 장”보다 “실제 게임 화면에서 무엇이 먼저 보이는가”다.

## 핵심 UX 원칙

Slay the Spire의 UI는 장식보다 의사결정 지원에 집중한다. 예쁘게 보이는 이유도 대부분 정보의 위치, 크기, 반복 패턴이 안정적이기 때문이다.

1. 모든 화면은 한 가지 결정을 중심으로 구성된다.
2. 정보는 사용되는 위치 근처에 놓인다.
3. 상시 정보와 순간 정보가 분리된다.
4. 텍스트는 카드/툴팁/선택지에 집중되고, 반복 정보는 아이콘화된다.
5. 위험은 행동 전에 보여준다.
6. 불가능한 행동은 조용히 막지 않고, 플레이어가 왜 안 되는지 즉시 알려준다.
7. 화면은 항상 “지금 무엇을 클릭해야 하는지”를 하나 이상 보여준다.

## 전체 화면 구조

### 상단 HUD

관찰:

- 전투, 지도, 보상 화면에서 상단 HUD는 계속 유지된다.
- 캐릭터 이름, 현재 HP, 골드, 유물/포션, 층 정보 같은 런 지속 정보가 작은 크기로 배치된다.
- HUD는 게임 세계 위에 얹힌 상태지만, 화면 주역을 빼앗지 않는다.
- 중요한 수치는 아이콘과 숫자로 짧게 보이고, 긴 설명은 hover tooltip로 처리된다.

UX 기능:

- 플레이어가 어느 화면에 있든 런 상태를 계속 기억하게 한다.
- 다음 선택의 비용을 즉시 계산하게 한다.
- “내가 지금 위험한가, 돈이 충분한가, 현재 빌드가 무엇인가”를 확인시킨다.

우리 게임 적용:

- `HP`, `Gold`, `Depth`, `Act`, `증표/서약 상태`, `보유 동료 수`는 상단 HUD에 유지한다.
- Seed, debug depth, raw id 같은 개발 정보는 일반 플레이 화면에서 숨긴다.
- 유대/서약은 전투에 직접 영향을 주는 순간만 강조하고, 상시 HUD에서는 작게 유지한다.

### 화면 배치 언어

관찰:

- 지도: 중앙은 경로, 오른쪽은 범례, 왼쪽 아래는 Return.
- 전투: 플레이어는 왼쪽, 적은 오른쪽/중앙, 손패는 아래, 에너지는 왼쪽 아래, End Turn은 오른쪽.
- 보상/카드 선택: 배경은 어둡게 죽고, 선택 UI가 중앙에 뜬다.
- 상세 카드 보기: 주변을 어둡게 만들고 확대 카드를 중앙에 둔다.

UX 기능:

- 화면마다 바뀌는 내용은 중앙에 있고, 반복 조작은 항상 비슷한 위치에 있다.
- 플레이어는 매 화면마다 새로 학습하지 않아도 된다.

우리 게임 적용:

- 전투 화면에서 개발자식 3분할 패널을 없애고, 캐릭터/적/손패가 실제 전장 위에 놓이도록 유지한다.
- 지도 화면에서 노드 텍스트 버튼을 쓰지 않는다. 아이콘과 경로가 먼저 보여야 한다.

## 지도 화면 분석

### 지도 배경

관찰:

- 지도는 양피지/스크롤처럼 보인다.
- 실제 배경과 분리되어 있어 정보면이 명확하다.
- 노드와 선이 배경 위에서 충분히 대비된다.

UX 기능:

- 지도는 전투 배경과 다른 정보 모드임을 즉시 알린다.
- 여러 경로를 한 화면에서 비교하게 한다.

우리 게임 적용:

- 괴물 재난 로드무비라는 설정에 맞춰, 양피지 대신 “작전서/계약 노선도/낡은 보급 지도” 느낌으로 간다.
- 배경 이미지는 분위기용이고, 실제 노드 가독성을 해치면 안 된다.

### 지도 노드

관찰:

- 노드는 텍스트 버튼이 아니라 아이콘이다.
- 지도 위치 종류는 아이콘으로 구분된다: Unknown, Merchant, Treasure, Rest, Enemy, Elite, Boss.
- 현재 선택 가능 노드는 밝은 테두리와 색으로 강조된다.
- 지나간 노드는 선택 불가능하지만, 경로 기록으로 읽힌다.
- 먼 노드는 어둡게 남아서 미래 계획용 정보가 된다.

UX 기능:

- 플레이어가 한눈에 위험/보상/회복/상점 밀도를 파악한다.
- “지금 갈 수 있는 곳”과 “나중에 가능한 계획”이 시각적으로 분리된다.

우리 게임 적용:

- 전투, 정예, 중간보스, 보스, 여관/모닥불, 상점, 이벤트, 동료 계약, 강화 노드는 모두 별도 아이콘을 가져야 한다.
- 각 노드는 라벨 텍스트를 노드 내부에 넣지 않는다.
- hover 또는 선택 시 하단 설명에 노드 이름, 위험도, 예상 보상을 표시한다.

### 지도 연결선

관찰:

- 경로는 점선이다.
- 모든 노드를 모든 노드와 연결하지 않는다.
- 이전 선택에 따라 다음 선택 가능성이 제한된다.
- 선택 가능한 다음 경로는 더 밝게 보인다.

UX 기능:

- 지도는 단순 목록이 아니라 전략적 약속이다.
- 플레이어는 “지금의 선택이 미래 선택지를 어떻게 줄이는가”를 본다.

우리 게임 적용:

- `current_depth + 1` 전체를 열면 안 된다.
- 이전 완료 노드의 `next_ids`에 포함된 노드만 선택 가능해야 한다.
- 선택한 경로는 따뜻한 색, 비활성 경로는 어두운 회색으로 둔다.

### 지도 범례

관찰:

- 범례는 오른쪽에 별도 패널로 존재한다.
- 노드 아이콘을 모르더라도 즉시 의미를 확인할 수 있다.
- 범례는 설명문이 아니라 아이콘과 한 단어 라벨 중심이다.

UX 기능:

- 초보자는 아이콘을 학습하고, 숙련자는 범례를 거의 보지 않는다.
- 지도 화면의 불확실성을 줄인다.

우리 게임 적용:

- 범례는 필요하다. 단, 장황한 설명 대신 `Fight`, `Elite`, `Camp`, `Shop` 같은 짧은 말로 충분하다.
- 세계관 용어를 쓰려면 처음에는 병기한다. 예: `Ally Contract`, `Upgrade`.

### 지도 상호작용

관찰:

- 선택 가능한 노드는 커서/하이라이트/테두리로 반응한다.
- 선택 불가능한 노드는 시각적으로 죽어 있다.
- 경로 선택은 보통 클릭으로 완료된다.

UX 기능:

- 플레이어가 잘못 클릭해도 혼란스럽지 않다.
- 선택 가능성이 명확하므로 튜토리얼 텍스트가 줄어든다.

우리 게임 적용:

- 클릭과 드래그 모두 지원하되, 기본 조작은 클릭이어야 한다.
- 선택 가능 노드는 hover 시 툴팁/하단 설명이 바뀌어야 한다.
- 잠긴 노드는 hover해도 “왜 못 가는지” 정도만 보여준다.

## 전투 화면 분석

### 전장 배경

관찰:

- 전투는 패널 안에서 벌어지지 않는다.
- 캐릭터와 적은 배경 위에 서 있고, UI는 필요한 곳에 얹힌다.
- 배경은 분위기를 주지만, 캐릭터/카드/의도를 방해하지 않는다.

UX 기능:

- 전투가 실제 장면처럼 느껴진다.
- 플레이어의 시선이 중앙 전장과 하단 손패 사이를 오가게 한다.

우리 게임 적용:

- 배경은 어둡고 안정적으로 두되, 전장 바닥과 캐릭터 실루엣은 분리되어야 한다.
- 적을 패널 카드처럼 가두기보다, 캐릭터 아래 HP/상태 UI를 붙인다.

### 플레이어 위치

관찰:

- 플레이어 캐릭터는 왼쪽에 고정된다.
- HP와 Block은 캐릭터 아래에 붙는다.
- 상태 아이콘은 캐릭터 주변 또는 HP바 근처에 표시된다.

UX 기능:

- 플레이어는 항상 자신의 생존 상태를 왼쪽 아래/중앙 아래에서 확인한다.
- 방어 수치가 적 공격 의도와 즉시 비교된다.

우리 게임 적용:

- 주인공 HP/Block/Energy는 큰 사이드 패널이 아니라 캐릭터 근처의 작은 HUD로 둔다.
- 동료가 있더라도 주인공 생존 정보가 가장 먼저 읽혀야 한다.

### 적 위치

관찰:

- 적은 오른쪽 또는 중앙 오른쪽에 배치된다.
- 적이 여러 명이면 가로로 나란히 서고, HP바와 이름이 각 적 아래에 붙는다.
- 적 정보는 적 개체에 붙어 있어 타깃 선택이 직관적이다.

UX 기능:

- “어떤 적에게 어떤 카드를 쓸 것인가”가 자연스럽다.
- 적 UI와 타깃 히트박스가 같은 위치에 있다.

우리 게임 적용:

- 적 패널을 크게 만들기보다, 적 스프라이트와 아래쪽 HP/Block/Mark를 묶는다.
- 적 intent는 반드시 적 머리 위에 있어야 한다.

### 적 Intent

관찰:

- Intent는 적 머리 위에 뜨는 아이콘이다.
- 공격 intent는 피해량과 공격 횟수까지 알려준다.
- 방어, 버프, 디버프, 공격+방어, 공격+디버프 등 복합 intent도 아이콘으로 구분된다.
- intent는 거의 신뢰 가능한 정보이며, 일부 효과만 예외를 만든다.

UX 기능:

- 플레이어는 무작위 피해가 아니라 예고된 위협에 대응한다.
- 턴의 재미가 계산과 선택으로 바뀐다.
- 방어카드, 공격카드, 디버프 카드의 가치를 즉시 비교하게 한다.

우리 게임 적용:

- 현재 `Intent: Claw Ledger (6 damage)` 같은 텍스트 칩은 임시다.
- 최종 전투 UI는 적 머리 위에 `공격 6`, `방어`, `치유 감소`, `버프`, `소환` 아이콘을 띄운다.
- 치유 감소 적처럼 우리 게임만의 핵심 위협도 intent 아이콘으로 먼저 읽혀야 한다.

### HP와 Block

관찰:

- HP는 빨간 바와 숫자로 표시된다.
- Block은 다른 색/아이콘으로 HP와 분리된다.
- 수치는 작지만 항상 정확하다.

UX 기능:

- 공격 결과를 예측할 수 있다.
- 적 처치 가능 여부와 방어 필요량이 한 화면에서 계산된다.

우리 게임 적용:

- 적 HP, Block, Tactical Mark는 같은 줄에 넣되, 마크는 아이콘화해야 한다.
- 플레이어 Block은 HP와 한 줄에 붙이되, 에너지와는 분리한다.

### 손패

관찰:

- 손패는 화면 하단에 부채꼴로 놓인다.
- hover된 카드는 위로 올라오고 커진다.
- 선택/드래그 중인 카드는 중앙 쪽으로 크게 뜬다.
- 손패는 전투 화면의 가장 큰 UI 영역이다.

UX 기능:

- 카드가 곧 행동이라는 점을 강하게 전달한다.
- 플레이어는 손패를 먼저 스캔하고, 그 다음 적 intent와 대조한다.

우리 게임 적용:

- 하단 손패가 항상 화면의 주인공이어야 한다.
- 카드가 많아져도 겹침은 의도된 팬 형태로 보여야지, 텍스트가 잘리면 안 된다.
- hover, select, drag, target highlight를 모두 넣어야 한다.

### 카드 구조

관찰:

- 카드 좌상단에 코스트가 있다.
- 카드 상단에는 이름 배너가 있다.
- 중앙에는 아트가 있다.
- 하단에는 타입과 룰 텍스트가 있다.
- Attack, Skill, Power는 프레임 형태와 색이 다르다.
- rarity는 이름 배너/프레임 색으로 구분된다.

UX 기능:

- 손패 스캔 순서가 고정된다: 비용, 이름/타입, 효과.
- 플레이어는 카드 전체 설명을 매번 다 읽지 않아도 된다.

우리 게임 적용:

- 카드 타입 색상을 명확히 한다.
- 동료 카드는 프레임/리본/증표 아이콘으로 주인공 카드와 구분한다.
- 긴 효과 텍스트는 카드 본문에서는 압축하고, 상세 보기에서 풀어야 한다.

### 카드 사용

관찰:

- 타깃이 필요한 카드는 드래그/선택으로 적에게 연결된다.
- 사용 중에는 카드가 크게 뜨고 대상 방향으로 시각적 연결이 생긴다.
- 타깃이 필요 없는 카드는 선택 즉시 발동할 수 있다.
- 에너지가 부족하면 말풍선으로 알려준다.

UX 기능:

- 실수 사용을 줄인다.
- 카드와 대상 사이의 관계가 물리적으로 느껴진다.

우리 게임 적용:

- 클릭 첫 단계는 카드 선택, 두 번째는 대상 선택으로 명확히 한다.
- 드래그 중에는 화살표/궤적선을 표시한다.
- 에너지 부족, 잘못된 대상, 죽은 대상 클릭은 짧은 피드백으로 처리한다.

### 에너지

관찰:

- 에너지는 왼쪽 아래 큰 원형 UI로 표시된다.
- 남은 에너지와 최대 에너지가 `3/3`처럼 즉시 읽힌다.
- 카드 코스트와 같은 영역 색/숫자 언어를 공유한다.

UX 기능:

- 이번 턴의 예산을 손패 바로 옆에서 계산한다.
- 카드를 쓸 수 있는지 판단하는 시간이 줄어든다.

우리 게임 적용:

- 에너지는 플레이어 사이드 패널에 묻히면 안 된다.
- 손패 왼쪽 아래 또는 플레이어 아래에 큰 원형 UI로 둔다.

### End Turn

관찰:

- End Turn은 오른쪽에 고정된다.
- 버튼은 전투 중 가장 명확한 비카드 행동이다.
- 턴 종료 버튼은 전장과 손패 사이에 있어 시선 이동이 짧다.

UX 기능:

- “더 할 행동이 없으면 여기”라는 안전한 종료 지점을 제공한다.

우리 게임 적용:

- End Turn은 하단 컨트롤바 속 작은 버튼이 아니라 전장 오른쪽 고정 버튼이어야 한다.
- victory 상태에서는 같은 위치가 Continue/Claim Reward로 바뀌어도 된다.

### 전투 로그

관찰:

- 일반 플레이 화면에 긴 전투 로그가 상시 노출되지 않는다.
- 피해, 방어, 상태 변화는 캐릭터 위 수치, 이펙트, 아이콘 변화로 전달된다.

UX 기능:

- 플레이어가 텍스트 로그를 읽지 않고도 턴 결과를 이해한다.
- 화면이 디버그 도구처럼 보이지 않는다.

우리 게임 적용:

- 전투 로그는 개발용이거나 접을 수 있는 보조 정보로 둔다.
- 최종 UI에서는 floating number, hit effect, status icon, toast를 중심으로 전달한다.

## 보상 화면 분석

### 전투 후 보상

관찰:

- 전투가 끝나면 배경 위에 보상이 모달처럼 열린다.
- 골드, 카드 보상, 포션, 유물 등이 개별 보상 항목으로 보인다.
- 카드 보상은 별도 선택 화면으로 들어간다.
- 보상은 클릭해 획득하는 물체처럼 보인다.

UX 기능:

- 전투와 보상 상태가 명확히 분리된다.
- 플레이어는 “이 전투에서 무엇을 얻었는가”를 단계적으로 확인한다.

우리 게임 적용:

- 전투 승리 후 바로 카드 선택으로 보내기보다, 보상 항목 리스트를 보여준다.
- 동료 유대, 카일 도박 진행도, 증표 보상도 보상 항목처럼 표시한다.

### 카드 보상

관찰:

- 카드 3장이 큰 크기로 펼쳐진다.
- 한 장을 고르거나 skip할 수 있다.
- 카드 hover는 상세 이해를 돕는다.

UX 기능:

- 덱빌딩 선택이 전투 보상 중 가장 큰 결정임을 보여준다.
- skip은 덱 압축 전략을 가능하게 한다.

우리 게임 적용:

- 보상 카드가 세 장이면 세 장이 동등한 시각 무게를 가져야 한다.
- 동료 카드 보상과 주인공 카드 보상은 시각적으로 구분한다.
- skip은 반드시 존재해야 한다.

### 유물/키워드 팝업

관찰:

- 유물 설명은 작은 아이콘에서 중앙 카드형 팝업으로 열린다.
- 주변은 어두워지고, 설명 카드만 집중된다.
- 키워드는 노란색/강조색으로 구분된다.

UX 기능:

- 화면을 떠나지 않고 상세 정보를 읽는다.
- UI 학습 비용을 tooltip과 popup에 분산한다.

우리 게임 적용:

- 증표, 서약 전술, 유대 보너스, 상태 이상은 모두 카드형 상세 팝업을 가져야 한다.
- 전문용어를 카드 본문에 전부 설명하지 말고 tooltip 계층으로 분리한다.

## 상점 화면 분석

### 상점 구성

관찰:

- Merchant/Shop은 카드, 유물, 포션, 카드 제거 서비스를 골드로 구매하는 화면이다.
- 상품은 종류별로 묶이고 가격이 즉시 보인다.
- 살 수 없는 항목은 회색/비활성으로 표시된다.
- 나가는 버튼은 명확하다.

UX 기능:

- 플레이어는 골드를 무엇으로 전환할지 비교한다.
- 가격과 상품 성능이 같은 화면에서 판단된다.

우리 게임 적용:

- 상점은 “카드/장비/소모품/제거” 구획을 나눈다.
- 가격은 항상 상품 카드에 붙인다.
- 골드가 부족하면 비활성화와 짧은 안내를 동시에 제공한다.

### 카드 제거

관찰:

- 카드 제거는 상품과 같은 경제 선택이다.
- 덱빌딩 게임에서 제거는 강력한 선택지이므로 별도 서비스로 보인다.

UX 기능:

- 좋은 카드를 추가하는 것과 나쁜 카드를 줄이는 것 사이의 전략 선택을 만든다.

우리 게임 적용:

- 우리 게임도 제거 서비스가 있다면 상점의 핵심 상품으로 취급한다.
- 삭제/정화/계약 파기 같은 세계관 명칭을 붙이되, 기능은 즉시 이해되어야 한다.

## 여관/휴식 화면 분석

### 휴식 선택

관찰:

- Rest Site는 기본적으로 Rest와 Smith 중 하나를 고른다.
- 선택지는 큰 아이콘과 짧은 설명으로 표시된다.
- 보유 유물에 따라 추가 선택지가 생길 수 있다.
- 보스 전에는 항상 휴식지가 있어 위험 대비 시간을 준다.

UX 기능:

- 체력 회복과 카드 강화의 긴장 관계를 만든다.
- 선택지가 적어서 한 번의 전략 결정을 선명하게 만든다.

우리 게임 적용:

- 여관은 `회복`, `강화`, `카드 제거/정비`, `동료 유대 이벤트` 중 2~3개만 보여준다.
- 모든 기능을 한 화면에 풀어놓지 않는다.
- 보스 전 휴식/정비 타이밍은 지도에서 미리 보이게 한다.

## 이벤트 화면 분석

### 이벤트 선택지

관찰:

- 이벤트는 이야기 텍스트와 선택지로 구성된다.
- 선택지는 비용/보상/위험이 명확히 적힌다.
- 선택할 수 없는 옵션은 조건 때문에 비활성화된다.

UX 기능:

- 랜덤 이벤트가 완전한 랜덤 피해가 아니라, 리스크를 읽고 감수하는 선택이 된다.

우리 게임 적용:

- 이벤트 텍스트는 짧고 선명해야 한다.
- 선택지에는 `얻음`, `잃음`, `확률`, `전투 발생` 같은 결과를 직접 쓴다.
- 증표 세계관 때문에 말장난/불리한 계약 이벤트를 만들 수 있지만, UI는 공정하게 결과를 알려야 한다.

## 카드 상세/덱 보기 분석

### 카드 확대

관찰:

- 카드 상세 보기에서는 한 장이 중앙에 크게 뜬다.
- 좌우 이동 버튼으로 다른 카드를 볼 수 있다.
- 키워드 설명이 옆에 작은 박스로 붙는다.

UX 기능:

- 작은 손패 텍스트를 보완한다.
- 키워드를 모르는 플레이어도 게임을 멈추고 이해할 수 있다.

우리 게임 적용:

- 모든 카드에는 상세 보기 모드가 있어야 한다.
- 상세 보기에는 `서약`, `유대`, `치유 감소`, `Tactical Mark` 같은 키워드 설명을 오른쪽에 붙인다.

### 덱/드로우/버림/소멸 보기

관찰:

UX 기능:

- 숙련자는 확률과 남은 카드를 계산한다.
- 초보자는 덱 구조를 학습한다.

우리 게임 적용:

- Draw, Discard, Exhaust는 숫자만 보이지 말고 클릭 가능해야 한다.
- 클릭하면 카드 그리드 모달이 열려야 한다.

## 상태/키워드/툴팁 분석

### 상태 아이콘

관찰:

- Strength, Dexterity, Vulnerable, Weak, Poison 같은 상태는 작은 아이콘으로 표시된다.
- hover하면 설명이 뜬다.
- 아이콘은 캐릭터/적 근처에 붙는다.

UX 기능:

- 복잡한 버프/디버프를 화면에서 압축한다.
- 읽고 싶은 순간에만 설명을 보여준다.

우리 게임 적용:

- 치유 감소, Tactical Mark, 유대 보너스, 서약 발동 대기 상태는 모두 아이콘화한다.
- 카드 설명문 안에 반복해서 긴 설명을 쓰지 않는다.

### 부정 행동 피드백

관찰:

- 에너지가 부족하면 “Not enough Energy” 말풍선이 뜬다.
- 잘못된 행동을 조용히 무시하지 않는다.
- 메시지는 짧고 화면 안 상황 위치 근처에 나온다.

UX 기능:

- 플레이어가 실패 이유를 즉시 학습한다.
- UI가 답답하게 느껴지지 않는다.

우리 게임 적용:

- 에너지 부족, 대상 없음, 이미 죽은 적, 선택 불가 노드는 모두 짧은 피드백을 띄운다.
- 전투 로그보다 현장 피드백이 우선이다.

## 시각 디자인 분석

### 색과 대비

관찰:

- 배경은 어둡고, 카드/HP/의도/버튼은 밝다.
- 강조색은 기능별로 다르다: HP는 빨강, 에너지는 노랑/주황, 상태는 아이콘 색, 선택 가능은 청록/밝은 테두리.
- 지도는 저채도 양피지 위에 어두운 선과 아이콘을 놓는다.

UX 기능:

- 중요한 행동 요소가 먼저 보인다.
- 수치 정보가 장식에 묻히지 않는다.

우리 게임 적용:

- 차가운 청회색과 모닥불색 팔레트는 유지하되, 전투 중요 정보는 더 밝은 대비를 준다.
- 후반 보라/진녹색 팔레트에서도 HP/에너지/위협 색은 변하지 않아야 한다.

### 텍스트 밀도

관찰:

- 지도 노드에는 텍스트가 없다.
- 전투 중 텍스트는 카드와 툴팁에 집중된다.
- 버튼 문구는 짧다: End Turn, Return, Skip.
- 장문의 설명은 모달이나 이벤트 화면에서만 허용된다.

UX 기능:

- 전투 속도를 유지한다.
- 플레이어가 화면 전체를 읽지 않고도 행동한다.

우리 게임 적용:

- 지도 노드에 `D4 combat Ransacked Camp Locked` 같은 텍스트를 넣으면 안 된다.
- 카드 본문은 1~2문장으로 압축한다.
- 긴 세계관 설명은 아트디렉터 문서와 이벤트 모달에 둔다.

### 애니메이션

관찰:

- 카드 hover, 카드 사용, 피해 숫자, 상태 변화, 보상 팝업이 움직인다.
- 애니메이션은 짧고 기능적이다.
- 사용 중인 카드가 커지는 것은 단순 장식이 아니라 선택 확인이다.

UX 기능:

- 플레이어의 행동이 시스템에 입력되었다는 감각을 준다.
- 결과가 어디에서 어디로 이동했는지 이해시킨다.

우리 게임 적용:

- 카드 드로우는 손패로 날아와야 한다.
- 카드 사용은 카드가 목표로 이동하거나 선을 그어야 한다.
- 피해/방어/회복은 floating number로 보여준다.

## 우리 게임에 옮길 때의 규칙

### 그대로 가져올 UX 문법

- 지도는 아이콘 노드와 점선 경로로 구성한다.
- 다음 선택 가능 노드만 밝게 표시한다.
- 전투는 배경 위에 플레이어/적/손패를 배치한다.
- 손패는 하단 부채꼴로 둔다.
- 적 intent는 머리 위에 둔다.
- 에너지는 손패 근처 큰 원형 UI로 둔다.
- End Turn은 오른쪽 고정 버튼으로 둔다.
- 보상, 카드 상세, 유물 상세는 배경을 어둡게 죽인 중앙 모달로 둔다.

### 우리 게임에 맞게 바꿀 요소

- 유물 자리는 `증표`, `장비`, `서약 효과`, `동료 유대 보너스`가 일부 대체한다.
- 지도 노드는 탑 등반이 아니라 재난지로 향하는 로드무비 경로다.
- 여관/휴식지는 모닥불, 임시 야영, 폐허 속 정비 장소로 표현한다.
- 동료는 별도 캐릭터가 아니라 전투 중 지원 카드/서약 효과/유대 보너스로 화면에 들어온다.
- Tactical Mark와 치유 감소는 Slay의 Vulnerable/Weak처럼 아이콘과 툴팁으로 학습시킨다.

### 피해야 할 안티패턴

- 전투 화면을 플레이어 패널, 중앙 패널, 로그 패널로 3분할하지 않는다.
- 지도 노드에 긴 텍스트를 넣지 않는다.
- 전투 로그를 최종 UI의 핵심 정보 전달 수단으로 쓰지 않는다.
- 개발용 버튼이나 Debug 화면을 일반 플레이 루프에 노출하지 않는다.
- 카드가 하단에서 잘려 룰 텍스트를 못 읽게 만들지 않는다.
- 정보 패널을 겹겹이 카드처럼 중첩하지 않는다.
- 적 intent를 텍스트 설명으로만 처리하지 않는다.

## 화면별 구현 체크리스트

### 지도

- 중앙 경로 패널이 화면 주인공인가?
- 노드는 아이콘 중심인가?
- 노드 내부 텍스트가 없는가?
- 현재 갈 수 있는 노드만 밝게 보이는가?
- 이전 선택에 따라 다음 노드가 제한되는가?
- 경로 연결선은 점선인가?
- 범례는 짧고 읽기 쉬운가?
- hover 시 노드 설명이 따로 표시되는가?
- 보스와 중간보스가 지도에서 미리 보이는가?
- 보스 전 회복/정비 타이밍이 보이는가?

### 전투

- 플레이어는 왼쪽, 적은 오른쪽/중앙인가?
- 손패가 하단의 가장 큰 UI인가?
- 에너지가 손패 근처에 크게 보이는가?
- End Turn이 오른쪽에 고정되어 있는가?
- 적 intent가 적 머리 위에 있는가?
- 적 HP/Block/상태가 적 근처에 붙어 있는가?
- 카드 hover가 확대/상승하는가?
- 카드 드래그/클릭 대상 지정이 명확한가?
- 에너지 부족 피드백이 현장 말풍선으로 나오는가?
- 긴 전투 로그가 화면을 차지하지 않는가?

### 카드

- 코스트가 좌상단에서 즉시 보이는가?
- 이름, 아트, 타입, 룰 텍스트의 위치가 일관적인가?
- Attack/Skill/Power가 색과 프레임으로 구분되는가?
- 주인공 카드와 동료 카드가 구분되는가?
- 긴 키워드는 툴팁으로 분리되는가?
- 카드 상세 보기가 있는가?

### 보상

- 승리 후 보상 목록과 카드 선택이 분리되어 있는가?
- 골드, 카드, 장비, 동료/유대 보상이 각각 물체처럼 보이는가?
- 카드 보상은 3장 선택과 skip을 제공하는가?
- 이미 획득한 보상은 상태가 바뀌는가?

### 상점

- 골드가 항상 보이는가?
- 상품 가격이 상품 옆에 붙어 있는가?
- 카드/장비/소모품/제거 서비스가 구획화되어 있는가?
- 살 수 없는 상품이 명확히 비활성화되는가?
- 나가기 버튼이 명확한가?

### 여관/이벤트

- 한 화면의 선택지가 너무 많지 않은가?
- 선택 결과가 명확히 적혀 있는가?
- 회복과 강화가 전략적 충돌을 만드는가?
- 이벤트 텍스트가 화면을 압도하지 않는가?
- 선택 불가 조건이 명확히 표시되는가?

## 결론

Slay the Spire의 UIUX에서 우리가 배워야 할 것은 특정 색감이나 카드 모양보다 “정보가 행동 위치에 붙는 구조”다. 지도는 미래 선택을 보여주는 전략 보드이고, 전투는 적 intent와 손패를 비교하는 판단 장치이며, 보상 화면은 덱을 바꾸는 의식이다.

우리 게임의 다음 UI 작업은 다음 순서가 맞다.

1. 지도: 실제 연결 경로, 아이콘 노드, hover 설명, 범례 완성.
2. 전투: 적 intent 아이콘, 에너지 원형 UI, 카드 hover/drag/target 강화.
3. 카드: 상세 보기와 키워드 tooltip.
4. 보상: 전투 보상 목록과 카드 선택 분리.
5. 상점/여관/이벤트: 선택 화면을 카드형 모달 체계로 통일.
